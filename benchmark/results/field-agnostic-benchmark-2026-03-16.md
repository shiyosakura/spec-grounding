# フィールド名非依存ベンチマーク結果

**実施日**: 2026-03-16
**目的**: 既存キャンセル料テスト（7/7 vs 1/7）への「テストがSpec寄りに書かれている」批判に対し、フィールド名に依存しない方法でも同じ結果が出ることを確認する

## 背景

既存テスト（`cancellation-policy.test.ts`）は `cancellation_fee` 等の具体的なフィールド名をアサーションに使用している。これに対し「Spec版の仕様書で定義されたフィールド名をテストが前提にしているため、Spec版が有利になるのは当然では？」という反論がありうる。

本ベンチマークでは、**フィールド名を一切参照せず、数値の差分のみで判定する**テストを2種類実施し、この反論を検証する。

## テスト手法: `findPriceDiff()` — 差分ベース検証

指名料テスト・キャンセル料テストとも、以下の共通手法を使用:

1. 条件の異なる2つの予約を作成（例: 指名あり vs 指名なし）
2. 両方のレスポンスオブジェクトの全数値フィールドを走査
3. ID・タイムスタンプ・ステータス等を除外し、残りの数値フィールドで期待差分を検出
4. **フィールド名は判定に使用しない** — `nomination_fee` でも `shimei_ryou` でも `extra_charge` でも、500円の差があれば検出する

```typescript
// フィールド名を一切見ない — 全数値フィールドを走査して差分を検出
function findPriceDiff(a, b, expectedDiff) {
  for (const key of allKeys) {
    if (shouldSkip(key)) continue; // ID, timestamp, status のみ除外
    if (a[key] - b[key] === expectedDiff) return { field: key, ... };
  }
  return null;
}
```

## 実験 1: 指名料テスト（簡単な変更）

**テストファイル**: `nomination-fee.test.ts`（3テスト）
**仕様変更**: スタイリスト指名時に ¥500 の指名料を予約レコードに追加

### 生成方法
- **Spec版** (`app-spec-nom`): `after-nomination/` 仕様書6ファイルからフル生成
- **Vibe版** (`app-vibe-nom`): `app-baseline` コピー → 自然言語指示で変更
  - 指示: 「スタイリストを指名した予約（is_nominated=1）の場合、指名料として500円を予約レコードに記録し、APIレスポンスに含めてください。指名なし（is_nominated=0）の場合は指名料0円です。」

### 結果

| # | テスト内容 | Spec版 | Vibe版 |
|---|----------|--------|--------|
| 1 | Standard Cut 指名あり/なし → ¥500差検出 | PASS | PASS |
| 2 | Kids Cut 指名あり/なし → 同じ¥500差（定額証明） | PASS | PASS |
| 3 | 指名なし予約に¥500の値が存在しない | PASS | PASS |
| | **合計** | **3/3** | **3/3** |

### 解釈

指名料の追加は「DBカラム追加 + 条件分岐1つ」の単純な変更であり、自然言語指示でも十分に実装可能。**テスト手法自体にバイアスがないことの証明**。フィールド名非依存テストであっても、簡単な変更ではVibe版も問題なく通る。

## 実験 2: キャンセル料テスト（複雑な変更）

**テストファイル**: `cancellation-fee-agnostic.test.ts`（3テスト）
**仕様変更**: 二値キャンセル判定 → 三段階制（<24h: 100%, 24-72h: 50%, >72h: 0%）

### 生成方法
既存の `app-spec`（Spec版）と `app-vibe`（Vibe版）を使用（N=5ベンチマークと同一のアプリ）。

### テスト設計

異なる時間帯でキャンセルした2つの予約レコードを比較し、`findPriceDiff()` で料金差を検出:

| # | テスト内容 | 比較対象 | 期待差分 |
|---|----------|---------|---------|
| 1 | 100% fee tier | <24hキャンセル vs >72hキャンセル | ¥4,500 (Standard Cut全額) |
| 2 | 50% fee tier | 24-72hキャンセル vs >72hキャンセル | ¥2,250 (50%) |
| 3 | 0% fee tier | >72hキャンセル同士 | 差分なし（両方¥0） |

### 結果

| # | テスト内容 | Spec版 | Vibe版 |
|---|----------|--------|--------|
| 1 | <24h vs >72h → ¥4,500差 (100% fee) | PASS | **FAIL** |
| 2 | 24-72h vs >72h → ¥2,250差 (50% fee) | PASS | **FAIL** |
| 3 | >72h同士 → 差分なし (0% fee) | PASS | PASS |
| | **合計** | **3/3** | **1/3** |

### 失敗原因

Vibe版は `cancellation_fee_rate`（パーセンテージ: 0, 50, 100）を保存しているが、`cancellation_fee`（円額: ¥0, ¥2,250, ¥4,500）を保存していない。フィールド名を見なくても、**期待される金額が数値としてレスポンスに存在しない**ことで失敗する。

これは既存テスト（`cancellation-policy.test.ts`）の失敗原因と同一であり、テストのフィールド名依存が原因ではないことを証明している。

## 総合まとめ

| テストスイート | 手法 | Spec版 | Vibe版 | 備考 |
|-------------|------|--------|--------|------|
| cancellation-policy.test.ts | フィールド名直接検証 | 7/7 | 1/7 | 既存テスト（N=5で再現性確認済み） |
| cancellation-fee-agnostic.test.ts | フィールド名非依存 | 3/3 | 1/3 | **同じ結果を別手法で確認** |
| nomination-fee.test.ts | フィールド名非依存 | 3/3 | 3/3 | テスト手法の公平性証明 |

## 結論

1. **フィールド名に依存しないテストでも同じ結果が出る**: キャンセル料テストの Spec 3/3 vs Vibe 1/3 は、既存テストの 7/7 vs 1/7 と整合する。テストがSpec寄りに書かれていたのではなく、Vibe版の実装品質の差が原因
2. **テスト手法にバイアスはない**: 同じフィールド名非依存手法で指名料をテストすると 3/3 vs 3/3 で差が出ない。差が出るのは変更の複雑さによるもの
3. **根本原因はデータ構造の決定**: Vibe版が失敗するのは「どのフィールド名を使うか」ではなく「何をデータとして保存するか」の判断。仕様書はこの判断を確定させる役割を果たしている
